# InfraPortal Platform API

Getting-started guide for the InfraPortal platform API. Created 2026-07-18 as part of v1.16.1 (OpenAPI Specs for All Services).

> **Runtime status:** all runtime endpoints have been offline since 2026-06-04, when the platform infrastructure was decommissioned to true zero (see `ROADMAP.md`). The OpenAPI specs indexed below document the API contract **as implemented in code**, verified by build + tests + CI, not against live endpoints. The example base URLs are historical.

---

## Platform overview

The platform is a CRM microservices system. The workspace in this repo contains 11 Rust service crates (axum 0.8 + sqlx 0.8 on PostgreSQL), fronted by a Go API gateway and authenticated by a Python auth service (both in separate repos).

| Service | Language | Purpose |
|---------|----------|---------|
| accounts-service | Rust | CRM accounts CRUD |
| contacts-service | Rust | CRM contacts CRUD, cross-service account validation |
| activities-service | Rust | CRM activities CRUD |
| automation-service | Rust | Workflow automation |
| integrations-service | Rust | Third-party integrations |
| opportunities-service | Rust | Sales opportunities CRUD |
| reporting-service | Rust | Saved reports and dashboard aggregation |
| search-service | Rust | Write-through search indexing and query |
| projects-service | Rust | Client portal projects |
| audit-service | Rust | Audit event ingestion and query |
| spend-service | Rust | Spend tracking |
| go-gateway | Go | API gateway: routing, rate limiting (separate repo) |
| auth-service | Python | Token issuance, JWKS, OAuth flows (separate repo) |

All CRM services expose the same surface shape: unauthenticated `/health` and `/ready` probes plus authenticated `/api/v1/<resource>` CRUD routes.

## Authentication

All `/api/v1` routes require a bearer JWT:

```
Authorization: Bearer <token>
```

- **Obtaining a token:** tokens are issued by auth-service (`POST /auth/login` with credentials, or `POST /auth/token` for service tokens). With the infrastructure offline there is no live issuer; for local development, sign an HS256 token with the service's `AUTH_JWT_SECRET`.
- **Validation:** services validate HS256 by default (`AUTH_JWT_ALGORITHM` also accepts HS384, HS512, RS256, RS384, RS512). Tokens must carry `exp` and an `iss` matching `AUTH_ISSUER` (default `auth-service`). The `sub` claim identifies the caller; `roles` is an array of strings, matched case-insensitively.
- **Admin role requirement:** the CRM services gate every `/api/v1` route on the `admin` role, not just on a valid token.

### 401 vs 403 semantics

| Status | Meaning | Error code |
|--------|---------|-----------|
| 401 | No `Authorization` header | `AUTH_REQUIRED` |
| 401 | Header is not exactly `Bearer <token>` | `AUTH_INVALID_FORMAT` |
| 401 | Signature, expiry, or issuer validation failed | `AUTH_INVALID_TOKEN` |
| 403 | Token valid but `roles` does not include `admin` | `FORBIDDEN` |

A 403 body looks like:

```json
{ "code": "FORBIDDEN", "message": "admin role required" }
```

## Error envelope

Every service-generated error response uses the same JSON shape:

```json
{
  "code": "VALIDATION_ERROR",
  "message": "name must not be empty",
  "details": { "field": "name", "constraint": "must not be empty" }
}
```

- `code` is machine-readable (`AUTH_REQUIRED`, `AUTH_INVALID_FORMAT`, `AUTH_INVALID_TOKEN`, `FORBIDDEN`, `VALIDATION_ERROR`, `NOT_FOUND`, `RATE_LIMITED`, `DB_ERROR`).
- `message` is human-readable.
- `details` is optional structured context; it is omitted entirely when absent, never null.

**Caveat:** rejections generated by the axum framework before a handler runs (400 for malformed JSON or invalid query string types, 422 for a JSON body that fails deserialization) have plain-text bodies, not this envelope. The per-service specs document these cases per operation.

## Rate limits

Rate limiting is enforced at the gateway (go-gateway, shipped in v1.10), per client IP per route, using a token bucket. Requests that bypass the gateway are not rate limited and carry no rate-limit headers.

Tiers are assigned by route prefix, not by HTTP method (a GET under a write-tier prefix is limited at the write tier):

| Tier | Route prefixes | Limit |
|------|----------------|-------|
| auth | `/api/auth` | 5 rps |
| write | CRM prefixes (accounts, contacts, opportunities, activities, automation, integrations, tasks, v1/projects) | 30 rps |
| read | reporting, search, events prefixes | 60 rps |
| default | any other proxied route | 15 rps |

Every gateway-proxied response carries:

- `X-RateLimit-Limit`: requests per second allowed for the matched tier
- `X-RateLimit-Remaining`: requests remaining in the current bucket
- `X-RateLimit-Reset`: Unix epoch second at which the bucket refills

When the limit is exceeded, the gateway returns 429 with a `Retry-After` header (seconds) and a `RATE_LIMITED` error body. Header interpretation, `Retry-After` handling, backoff strategies, and copy-paste retry snippets are covered in the [rate limiting adoption guide](RATE_LIMITING.md).
