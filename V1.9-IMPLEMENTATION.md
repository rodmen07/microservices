# v1.9 Implementation Guide: Distributed Tracing & Observability

## Overview

v1.9 adds end-to-end request tracing across all 11+ microservices using OpenTelemetry and GCP Cloud Trace. Requests flow through go-gateway with a W3C traceparent header, which is propagated through all downstream services.

## Architecture

```
Client Request
    ↓
go-gateway (middleware/traceparent.go)
    - Extracts or generates W3C traceparent header
    - Propagates to all downstream services
    ↓
Rust Service (main.rs)
    - Initializes OpenTelemetry tracer with Cloud Trace exporter
    - Receives traceparent via http.Header.Get("traceparent")
    ↓
Handler (#[instrument] macro)
    - Spans automatically created for each handler
    - Nested spans for database queries
    ↓
GCP Cloud Trace UI
    - Complete request trace visible with latencies
    - Correlates logs via trace ID
```

## Pattern for Each Rust Service

### Step 1: Update Cargo.toml Dependencies
Add these lines to the `[dependencies]` section:
```toml
opentelemetry = { version = "0.23", features = ["trace", "metrics"] }
opentelemetry-gcp = { version = "0.23" }
opentelemetry-http = "0.12"
opentelemetry-semantic-conventions = "0.15"
tracing-opentelemetry = "0.24"
```

Also update `tracing-subscriber` to include `json` feature:
```toml
tracing-subscriber = { version = "0.3", features = ["env-filter", "fmt", "json"] }
```

### Step 2: Update main.rs
Replace the tracing initialization with:
```rust
use tracing_subscriber::layer::SubscriberExt;

#[tokio::main]
async fn main() {
    // Initialize OpenTelemetry tracer for Cloud Trace
    let tracer = match opentelemetry_gcp::CloudTraceExporter::new()
        .install_batch(opentelemetry::runtime::Tokio)
    {
        Ok(t) => Some(t),
        Err(e) => {
            eprintln!("Failed to initialize OpenTelemetry: {}", e);
            None
        }
    };

    let tracing_layer = if let Some(t) = tracer {
        Some(tracing_opentelemetry::layer().with_tracer(t))
    } else {
        None
    };

    let subscriber = tracing_subscriber::fmt()
        .with_env_filter(
            tracing_subscriber::EnvFilter::try_from_default_env()
                .unwrap_or_else(|_| "<service_name>=info,tower_http=info".into()),
        );

    if let Some(layer) = tracing_layer {
        tracing::subscriber::set_default(subscriber.with(layer).finish());
    } else {
        tracing::subscriber::set_default(subscriber.finish());
    }

    // ... rest of main.rs
}
```

### Step 3: Add #[instrument] Macro to Handlers
In each handler file (e.g., `handlers/accounts.rs`), add the import:
```rust
use tracing::instrument;
```

Then decorate each handler function:
```rust
#[instrument(skip(state, headers), fields(request_id = %uuid::Uuid::new_v4()))]
pub async fn list_accounts(
    headers: HeaderMap,
    State(state): State<AppState>,
    Query(params): Query<ListAccountsQuery>,
) -> Response {
    // ... handler body
}
```

The `skip` parameter excludes large objects from tracing to reduce noise.

### Step 4: Ensure HTTP Client Propagates traceparent
When using `reqwest` for inter-service calls, ensure traceparent is propagated:
```rust
// In audit emit or other inter-service calls:
if let Ok(tp) = std::env::var("TRACEPARENT") {
    request = request.header("traceparent", tp);
}
```

However, since traceparent is in request headers, a better pattern is to pass it through the handler context or use middleware.

## Testing Locally

1. Disable Cloud Trace export (will use stdout):
```bash
export RUST_LOG=info
cargo run
```

2. Make a request through go-gateway:
```bash
curl -i http://localhost:8080/api/accounts/api/v1/accounts
```

3. Check logs for traceparent header:
```
traceparent: 00-<trace-id>-<span-id>-01
```

## Testing in Staging/Production

1. Ensure GCP Cloud Trace API is enabled:
```bash
gcloud services enable cloudtrace.googleapis.com --project=microservices-489413
```

2. Grant service account access:
```bash
gcloud projects add-iam-policy-binding microservices-489413 \
  --member="serviceAccount:microservices@microservices-489413.iam.gserviceaccount.com" \
  --role="roles/cloudtrace.agent" \
  --condition=None
```

3. Deploy service and make requests

4. View traces in Cloud Console:
```
https://console.cloud.google.com/traces/list?project=microservices-489413
```

## Common Issues & Solutions

### Issue: OpenTelemetry initialization fails
**Solution:** Check that service account has `roles/cloudtrace.agent` IAM role. Falls back to stdout logging.

### Issue: Traceparent header not propagated
**Solution:** Ensure go-gateway middleware is in the chain and proxy.go explicitly sets the header.

### Issue: High latency spike after deployment
**Solution:** Normal - GCP Cloud Trace export can add 10-50ms per request. Tune batch exporter settings if needed.

### Issue: No spans appearing in Cloud Trace
**Solution:** 
1. Check service account has correct IAM role
2. Verify `GOOGLE_APPLICATION_CREDENTIALS` is set
3. Check Cloud Trace API is enabled
4. Review logs for OpenTelemetry errors
